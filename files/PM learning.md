​	**Speed, meet beauty.** 独立思考，分析问题，逻辑推理，

## 为什么要精益产品探索

病毒式扩散之前，需要跨越鸿沟。

从天使用户到初期用户的跨越。

找到天使用户：

1. 存在问题或痛点
2. 知道他们自己存在问题或痛点
3. 尝试寻找解决方案
4. 倒腾他们自己的解决方案
5. 愿意并且能够为解决方案付钱

天使用户的特征：

1. 用户确认问题存在
2. 用户不会接受诸如“这就是生活”或“它超出了我的控制”等理由
3. 用户已经投入资源（时间、金钱、学习）去试图解决这个问题
4. 用户已经改变行为去尝试解决这个问题。
5. 用户没有受到去解决这个问题的任何限制。

你的目标是天使用户，不是主流用户。

客户根本不在乎你的解决方案，他们只关心遇到的问题。

##精益验证板：场景与用户

用户>>场景>>用户>>问题>>问题的假设>>解决方案

empathy（一秒钟变傻瓜/移情）>>define（问题/痛点）>>ideate（方案/功能）>>prototype（用户/场景）>>test（天使用户/主流用户）

假设>>验证>>访谈>>验证>>实施>>验证>>迭代>>验证

（验证=技术发烧友、天使用户、初期用户、后期用户）

用户：

1. 先考虑你的用户
2. 用户被验证后才考虑你的客户
3. 站在你的用户角度去思考问题
4. 不管商业模式多么复杂，不管用户是免费还是收费，都需要最终用户使用你的产品
5. 用户细分需要足够细，无论是红海还是蓝海，都需要努力找到那一个小的细分人群验证你的商业模式
6. 用户人群的描述至少需要细致到可以找到他们（比如你的亲人）
7. 先发散，逐一验证或推翻
8. 如果发现问题无法聚焦，则退回到人群细分

最好的技术架构和需求是，自组织团队中出来的。（技术、运营、销售、产品）

场景：

1. 移动互联网产品的场景感最强
2. 有的产品可能第一步考虑的是场景，接下来是场景下的用户细分
3. 先对场景进行假设，然后实地验证对场景的描述
4. 场景不同，问题和需求不同
5. 场景不同，用户所处的背景条件不同

场景可能是首先要考虑的

问题：

1. 用户和场景清晰后，抛弃掉脑子中所有的点子和创意
2. 每一个最终用户在每一个场景下所遇到的问题
3. 不是解决方案
4. 只有正确识别用户的问题，才能为用户提供解决方案
5. 可以采用五个为什么进行逻辑推理
6. 没有找到问题的根源，只会治标不治本
7. 如果问题不具体，多数是用户人群描述和定位模糊，或用户人群太泛，太大，需要对用户人群
8. 多数的问题都和情感有关

比如：这个工件为什么尺寸不合格？因为装夹松动；

1. 为什么装夹松动？因为操作工没装好；

2. 为什么操作工没装好?因为操作工技能不足；

3. 为什么技能不足？因为人事没有考评

4. 类似这样的情况，在5Why分析中，经常发现。

   所以，我们在利用5Why进行根本原因分析时，一定要把握好一些基本原则：

   1. 回答的理由是受控的；
   2. 询问和回答是在限定的一定的流程范围内；
   3. 从回答的结果中，我们能够找到行动的方向。

## 问题背后的假设

什么是问题背后的假设：

1. 组成这个问题的各个可度量的要素
2. 实例剖析：
   1. 问题/痛点:上班路途遥远
   2. 假设一:时长-上班路途遥远花费时长
   3. 假设二：交通工具
   4. 假设三：发生的日期
3. 假设尽可能可度量的指标
4. 假设不同，解决方案完全不同
5. 哪些假设，需要验证

--

最低的成本验证最高的假设的方法：

###客户访谈：

1. 你是否遇到过这个问题
2. 告诉我你上次遇到问题的情况和故事
3. 对你来说，解决这个问题的最佳方案是什么？

--

## 解决方案：

1. 收集所有被验证后的假设，形成解决方案
2. 运用迭代式开发，迅速开发出MVP，交付给天使用户
3. 快速反馈，在功能中埋点，收集用户的所有使用习惯和行为。最后验证对用户的所有假设
4. 从反馈的数据学习，调整下一步的产品功能和策略
5. 对于产品的任何一个功能/故事，反思这个功能可以解决【用户】在【场景】下的【什么问题】

--

什么是MVP：minimal viable product ，是推翻你最高风险假设的试验品。

1. 问题探索：是否存在？谁？
2. 产品推介：推广渠道
3. 产品适配：交付客户体验

--

成功的创业公司都是在资源耗尽之前，进行足够次数的迭代。

## 商业模式画布

系统的反应你的商业模式：

四个视角：

1. 如何提供（基础设施）（重要伙伴-关键业务+核心资源）
2. 提供什么（提供物）（价值主张）
3. 为谁提供（客户）（客户细分-客户关系+渠道通路）
4. 成本收益多少（财务）（成本结构-收入来源）

--

精益画布 lean canvas   VS  商业模式画布

1. 独特卖点---价值主张
2. 门槛优势---客户关系
3. 关键指标---核心资源
4. 问题      ---重要伙伴

--

精益画布 lean canvas

1. 问题--客户最需要解决的三个问题
2. 解决方案--产品最重要的三个功能
3. 关键指标--应该考核哪些东西
4. 独特卖点--用一句话简明扼要但引人注目的话阐述为什么你的产品与众不同，值得购买
5. 门槛优势--无法被对手轻易复制或买去的竞争优势
6. 渠道--如何找到客户
7. 客户群体分类--目标客户
8. 成本分析--争取客户所需花费，销售产品所需花费，网站架设费用，人力资源费用
9. 收入分析--盈利模式，客户终身价值，收入，毛利

--价值创造需要基础设施，基础设施需要成本投入。

# 产品的道、法、术、

互联网产品的道--哲学层

互联网产品的法--战略层：做正确的产品

互联网产品的术--战术执行层：正确的做产品

正确的互联网产品，在正确的时间，正确的做出来

## 道

哲学层：本质、规律

用户与市场是检验产品价值的唯一标准

产品价值量守恒：

使用价值=商业价值

产品复杂度守恒

让用户的交互行为变得简单，那么隐藏在产品背后的复杂性就增加了。

靠谱的商业模式：一句话可以说清楚

## 法

战略层：做正确的产品

1. 感性和理性的结合：需求采集和市场分析
2. 行业和市场的结合：洞察需求与满足需求 （从行业端去入手，市场端微观控制）
3. 理想和现实的结合：战略景愿与产品定位

--

例如：360安全卫士

杀毒不等于安全，用户要的不是杀毒，而是安全，包括数据安全，账号安全，网购安全下载安全等

例如：慢病管理

测血压不等于措施，用户要的不是测血压，而是措施有没有效果，包括饮食，服药，避免并发症，（比如高压为什么这么高，低压为什么这么低，服药了有没有效果，需不需要去医院，需不需要服药，是否能不吃药食疗，有没有并发症，需要注意什么，）----医生的科普视频教程更容易被接受

## 术

产品目标优先，业务规则次之，产品元素最后

用户体验的要素：

1. 战略层：产品目标，用户需求
2. 范围层：功能说明，内容需求
3. 结构层：交互设计，信息架构
4. 框架层：界面设计，导航设计
5. 表现层：视觉设计

--产品目标>>功能需求>>交互流程>>UI界面>>视觉效果

启示：真正搞清楚市场、用户真正需要什么样的产品。这比拼UI,UX更靠谱

--

准确把握用户需求：

举个栗子：客户需求是：我想要一匹更快的马。

如果场景是：客户想去北京

那么你就可以给客户一辆汽车，更好的满足需求。

如果场景是：客户有很多汽车，想要去赛马

那么你就不是给汽车而是一匹汗血宝马了。

产品也讲辩证法--舍得

--启示:要想让用户喜欢我们的互联网产品，一定要将我们的互联网产品设计的像一个举止得体的人。（拟人化）

## 需求的分析与管理

什么是需求？

从用户的角度：我要

​	喝水/一把扇子/开窗户/洗冷水澡

背景：

​	用户呆在屋子里，汗流浃背，非常热

从产品经理的角度:我给

​	给用户解决"这里太热了"

--启示：需求就是用户的最终目的，他不是功能，不是解决方案，只与最终目的有关

一个很简单的需求：

我想告诉你一件事>>

从古至今的解决方案是：

​	喊>>驿站传书>>飞鸽传书>>邮局寄信>>发电报>>手机通话>>上网聊QQ>>手机聊微信

--启示： 用户的最终目的，也就是需求几乎是不会改变的。

找准需求的重要意义：

案例1：QQ聊天在不断演变，（终端，功能，网络环境）

--用户需求不变，但是解决方案及用户的期许一直会变。

案例2：诺基亚，推出全屏手机-销售额不大以为需求量不大，然而苹果推出全屏手机颠覆市场。

--用户对产品投了反对票，不代表对他的需求投了反对票。

--启示：应该围绕需求优化解决方案，而不是围绕解决方案优化解决方案。（走出去看需求，不是在办公室看解决方案）

举个栗子：客户需求是：我想要一匹更快的马。

如果场景（背景，限制）是：客户想去北京

那么你就可以给客户一辆汽车，更好的满足需求。

如果场景（背景，限制）是：客户有很多汽车，想要去赛马

那么你就不是给汽车而是一匹汗血宝马了。

--

分析需求的利器---马斯洛层次需求（新网络需求：装逼,怕死,爱美）

1. 生理需求
2. 安全需求
3. 社交需求
4. 尊重需求
5. 自我实现

--越底层越原生态，需求越强壮。先满足客观需求再满足主观需求。

所有需求都可以归类转化：用户需求>>产品需求>>马斯洛需求

例子：点券转账

用户需求：

​	虚拟货币交易或赠送

产品需求：

​	账户之间的点券转账

马斯洛需求：

​	社交，感情化需求

## 准确找到需求

用户的需求像冰山

明确的需求>>隐含的需求>>未知的需求

言，行，想，（在ta身边，为ta设计，找到主流用户的痛点）

例子：索尼做了用户调查，询问喜欢什么颜色的随身听，大部分用户说喜欢黄色，最后让用户自己选择赠品的时候，大部分用户选了黑色。

需求的分类：

受众大小，使用频度，依赖程度，

真需求 VS 假需求：

例子1：首页 快速充值>>用户没有安全感，没有人使用

什么是错误的设计

1. 失败的心智模型：：比如：卡罗曼双人自行车（情侣车）--双人背靠背
2. 失败的预设用途：：比如：看上去像拉实际上应该是推的门把手（麦当劳只能推，没有门把手）
3. 没有正确的显示：：比如：将页面中次要按钮高亮显示，而将主按钮普通显示
4. 没有有效的反馈：：比如：吹风机的呼呼声音，不是噪音哦

--

如何避免设计中的错误

1. 心智模型是指用户在思考如何使用产品的过程中不会基于产品经理提供的设计，而是出自用户自身对产品的任何符合自身习惯的行为方式。
2. 反馈：用户额每一项操作必须得到及时明显的反馈（利用声音和可行性进行反馈）
   1. 将看不见的部位显示出来
   2. 设计合理的显示界面
   3. 给出动态的反馈效果
